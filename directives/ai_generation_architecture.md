# Amirani AI Architecture & Logic Directive

This directive outlines the technical architecture, data flow, structure, and unification standards for both AI-generated and Trainer-assigned Diet and Workout plans within the Amirani platform, capturing 100% of the entire macro and micro processing boundaries.

---

## 🏗️ 1. Conceptual Unification

To bridge the gap between Mobile, Backend (Prisma), and AI generation, we use a distinct **Master Template -> Assignment** pattern.
Whether generated by a Human Trainer or the AI, the resulting database footprint is mathematically identical.

### 1.1. Unified Naming Conventions

#### Root Data Structures
- **`MasterDietTemplate` / `MasterWorkoutTemplate`**: The core blueprint. For AI, it's generated dynamically. For trainers, it's manually assembled.
- **`DietPlan` / `WorkoutPlan`**: The actual "Assignment". This links a `User` to a `MasterTemplate`. It holds the `startDate`, `status: ACTIVE | ARCHIVED`, and the 4-week scope.

#### Diet Specific Structures
- **`MasterMeal`**: Represents a meal slot (e.g., "BREAKFAST") inside a template. Enums strictly regulate day definitions (`MONDAY` - `SUNDAY`). (Mobile: `PlannedMealEntity`)
- **`MasterMealIngredient`**: Specific nutritional data (calories, macros, amount). To prevent payload bloat, macros are calculated at the line level. (Mobile: `IngredientEntity`)

#### Workout Specific Structures
- **`MasterRoutine`**: A specific day's session (e.g., "Push Day"). (Mobile: `DailyWorkoutPlanEntity`)
- **`MasterRoutineExercise`**: Contains sets, reps, and mechanics. (Mobile: `PlannedExerciseEntity`, `ExerciseSetEntity`)

### 1.2. Database Intimacy (Schema Relations)
- **Assignments are 1-to-Many**: A `User` can have many `DietPlan` assignments over their lifetime, but only one is `ACTIVE`.
- **Historical Integrity**: The schema resists cascading deletes. The template structure remains intact so that `DailyProgress` tracking logs are never orphaned if a Trainer deletes a template.

## 📝 2. User Profile & Context Information

When generating an AI plan, the prompt compiler (`queue.ts` via `api_strategy.dart`) merges two sources of information to personalize the output:

### 2.1. Information Pulled from the User Profile & Localization
The platform automatically injects chronic data points to shape the baseline logic:
- **Metrics**: Weight (kg/lbs), Height (cm/in), Age, Gender
- **Localization (EU vs US Standards)**: The backend strictly looks at the unit format the member used when onboarding their height and weight. 
  - If they chose **lbs/in**, the system dictates the AI and Trainer interfaces to return Imperial ingredients (`oz`, `lbs`, `cups`, `fl oz`).
  - If they chose **kg/cm**, the system defaults to Metric assignments (`g`, `kg`, `ml`).
- **Health**: Target Weight (kg/lbs), Medical Conditions, persistent Injuries
- **Basics**: BMR / TDEE (Total Daily Energy Expenditure) 
- **Limits**: Target Calories, Target Proteins (Calculated internally based on the user's goals prior to generation)

### 2.2. Information Defined During Creation (Onboarding Form)
The user selects dynamic preferences just before generating:
- **Workout Preferences**: Goal (e.g. Muscle Gain, Weight Loss), Fitness Level, Training Split (e.g. Bro Split, Push/Pull/Legs), Preferred Training Days, Available Equipment, Target Muscles.
- **Diet Preferences**: Dietary Style (e.g. Vegan, Keto), Budget, Disliked Foods (specifically excluded), Allergies with Severities (e.g. Peanuts - Severe), Meals per day (e.g. 3, 5), Max Prep Minutes, Specific Meal Times (e.g. Breakfast at 08:00).

## 🤖 3. The AI Generation Pipeline (100% End-to-End)

### Step A: The Mobile Request
1. Mobile calls `POST /api/sync/ai/generate-plan`. Included in the payload are the merged Profile and Onboarding data points.
2. If using `ApiGenerationStrategy`, the backend evaluates the user's SaaS tier (`UserTierLimits`) to verify they have sufficient AI Generation Credits. (e.g., deducting 1 credit for Diet or 2 for Both).
3. Backend enqueues a background job via BullMQ (`queue.ts`) and returns `HTTP 202 Accepted` with a `jobId`.

### Step B: The AI Polling
1. Mobile begins a long-poll (`_pollJobStatus`) to GET `/api/sync/ai/status/:jobId`.
2. It uses **exponential backoff** (immediate first check → 2s → 4s → 8s → 10s cap, max 60 attempts ≈ 5 minutes total) to receive live progress updates (e.g., *20% - Building Profile*, *80% - Hydrating Ingredients*). The first check fires with zero delay — subsequent waits grow exponentially. This prevents polling storms under load.

### Step C: Backend Job Processing & AI Provider (`queue.ts`)
1. **Prompt Engineering**: The backend parses all constraints and sets a system prompt commanding pure JSON generation.
2. **Provider Call**: The request goes to OpenAI, Anthropic, or DeepSeek based on the Active `PlatformConfig`.
3. **Usage Limits**: The `maxTokensPerRequest` is dynamically fetched from central configuration to prevent over-billing.

### Step D: Backend Macro Strict-Validation
Before the AI's JSON is even trusted to be saved:
- The backend iterates over every Meal and sums the protein, carbs, fats, and calories.
- If the AI hallucinated "Ghost Calories" (e.g., 50g protein + 50g carbs = 400 calories, but the AI accidentally outputted 800 total calories), the validation engine intercepts this mathematical anomaly and corrects the master `planMeta` to perfectly reflect accurate caloric math.

### Step E: Persistence & Safety
1. **Template Creation**: The `Master...Template` and child tables (`MasterMeal`, etc.) are created.
2. **Rollback Safety**: An active `DietPlan` or `WorkoutPlan` is ONLY created at the very end.
3. **Deactivation**: Just before finishing, any previously ACTIVE plans for the user are marked `status: ARCHIVED`.

### 3.6. Zero-Hallucination Loop & Safety Constraints
To enforce absolute perfection and protect the mobile app from AI hallucinations, the backend acts as a **Deterministic Authority**. 

#### 3.6.1 The Safety Tolerance & Schema Matrix
Before accepting the output, the backend enforces rigid bounds over the AI via `collectWorkoutErrors()` / `collectDietErrors()` in `queue.ts`. These functions return a full list of all errors at once (rather than throwing on the first one), enabling the repair loop to fix them in a single targeted re-prompt.

**Diet validators:**
- `planMeta.dailyCalories` must be in range **800–6000 kcal**.
- **Macro formula**: `protein × 4 + carbs × 4 + fats × 9` must equal `dailyCalories` **±5%**. This is the single most important guard against Ghost Calories.
- Per-meal calories must be **300–900 kcal**. Per-meal protein must not exceed **70g**.
- Every meal must have at least one ingredient. Ingredient-level calories must not all be zero.
- Minimum **4 days** required (the 4 foundational ABCD days before mirroring: MONDAY, TUESDAY, WEDNESDAY, SUNDAY).

**Workout validators:**
- Exercises per set: **2–6 sets** per exercise. Reps: **3–20** per rep range.
- Total sets per day must not exceed **35**.
- Maximum **6 active training days** per week. At least **1 REST day** required.
- Maximum **28 days** total (the 4-week window).

#### 3.6.2 Zero-Wait Fallback & The Self-Healing Repair Loop
Three-tier protection ensures the user always receives a plan:

**Tier 1 — 8-second AI hard cap (`Promise.race`):**  
The AI call races an 8-second timeout. If the timeout wins (AI too slow or network error), execution jumps directly to Tier 3.

**Tier 2 — 1-shot targeted repair (15-second cap):**  
If the AI responds within 8 seconds but the plan fails validation, the backend runs `collectWorkoutErrors()` / `collectDietErrors()` to gather every error into a list, then sends a **structural summary** (not raw JSON) to the AI: *"Fix ONLY these issues."* Temperature is lowered to 0.3 for deterministic correction. This repair call has its own 15-second hard cap. **Exactly one repair attempt is made** — never more.

**Tier 3 — Deterministic fallback (`buildFallbackPlan`):**  
Triggered if: AI timed out (Tier 1), or the repair call timed out / returned invalid JSON, or the repaired plan still fails validation. The fallback generates a structurally valid plan from hardcoded templates (bodyweight workout or balanced **4-day ABCD** diet) that passes all validators without any AI involvement. The job still completes with `status: COMPLETED` so the user is never left with a spinner.

#### 3.6.3 Output Versioning
The system prompts include a `promptVersion: "v1"` / `schemaVersion: "v1"` example in the schema block to signal expected format to the AI. Version fields are currently advisory — the validation layer does not reject plans that omit them. Full enforcement (rejecting plans with mismatched schema versions) is a planned upgrade for when the schema makes a breaking change.

## 📅 4. Token Shields, Days, and Expected Formats

We apply strict token optimization to ensure AI providers don't randomly truncate large JSON responses.

### 4.1. Diet Generation (The "Mirror" Pattern) & Meal Count Mapping
- **Meal Count Rules**: The backend dynamically assigns explicit meal types based on the user's requested `mealsPerDay` count.
  - **2 Meals**: `BREAKFAST`, `DINNER`
  - **3 Meals**: `BREAKFAST`, `LUNCH`, `DINNER`
  - **4 Meals**: `BREAKFAST`, `LUNCH`, `SNACK`, `DINNER`
  - **5 Meals**: `BREAKFAST`, `SNACK 1`, `LUNCH`, `SNACK 2`, `DINNER`
- **AI Prompt Limit**: The AI is asked to generate exactly **4 Unique Foundational Days** using explicit canonical day names (`MONDAY`, `TUESDAY`, `WEDNESDAY`, `SUNDAY`). We **strictly forbid** abstract names like `DAY_1` or `Day 2` to prevent mapping collisions. Sunday is a standalone day because it only appears once in the 7-day cycle; the other 3 days each appear twice.
- **Per-Ingredient Macros**: Every ingredient MUST include `calories`, `protein`, `carbs`, `fats` as integers. The backend uses these to recalculate meal totals (Ghost Calories fix). Ingredient macro omission is a hard validation failure.
- **Backend Mirroring (ABCD pattern)**: When parsing, the backend expands the 4 foundational days into a full 7-day week, covering every possible plan start day:
  - Day A (`MONDAY`) → also populates `THURSDAY`
  - Day B (`TUESDAY`) → also populates `FRIDAY`
  - Day C (`WEDNESDAY`) → also populates `SATURDAY`
  - Day D (`SUNDAY`) → standalone (appears only once)
- **Rolling 28-day window**: Since all 7 day-of-week slots are covered, the plan start day is irrelevant. A member generating on Thursday gets a Friday start (TUESDAY template) and a Monday end (MONDAY template) — the mobile maps by calendar weekday, not by plan offset.

### 4.2. Workout Generation (The "Wave Periodization" Pattern)
- **AI Prompt Limit**: The AI generates exactly **7 Unique Days** (Monday to Sunday), including rest days.
- **Mobile Processing**: The mobile app receives the 7-day structure. Rather than generating 28 distinct days, the Flutter frontend (`_applyProgressiveOverload` in `ai_orchestration_service.dart`) applies **Wave Periodization** across 4 weeks. The exact formulas per goal:

| Week | muscleGain | weightLoss | endurance | strength | General |
|------|-----------|------------|-----------|----------|---------|
| 1 | Base | Base | Base | Base | Base |
| 2 | `reps + 2` | `reps + 2` | `reps + 3` | `reps × 0.85` | `reps + 2` |
| 3 | `reps + 4`, `sets + 1` | `reps + 4` | `reps + 6` | `reps × 0.70`, `sets + 1` | `reps + 3` |
| 4 | `reps` (deload to base) | `reps` (deload to base) | `reps` (deload to base) | `reps` (deload to base) | `reps` (deload to base) |

- All values are rounded to the nearest integer. Rest days are never modified.
- This logic runs entirely on the mobile device — no extra AI call needed for 4 weeks of progressive plans.

## 🧑‍🏫 5. Human Trainer Assignment vs AI

The system is designed so the mobile device doesn't care whether AI or a Human created the plan. They are fundamentally indistinguishable on the Client.

1. **AI Flow**:
   - `isAIGenerated: true`
   - Master Templates are bound only to the user requesting them.
2. **Trainer Flow**:
   - `isAIGenerated: false`
   - Trainers build a library of `MasterDietTemplate` objects in their dashboard.
   - A Trainer goes to a User's profile -> "Assign Plan".
   - The backend attaches the user to the Trainer's Master Template, creating a new `DietPlan` instance.
3. **Push Notifications (Triggering Sync)**:
   - When a trainer assigns a plan, the backend fires a Push Notification triggering the mobile app to silently fetch the newest assignment via `/sync/down`.

## 📱 6. Mobile Interaction, Functionality & Offline Lifecycle

### 6.1. Mobile Storage, Sanitization, and Capacity
1. **Network Sanitization**: When `/sync/down` completes, the JSON payload hits the `ai_orchestration_service.dart`. This layer meticulously sanitizes the data:
   - Null checks (missing descriptions become empty strings).
   - Numerical casting (making sure strings like `"200"` for calories are forced back to integers).
2. **Local Storage (Hive Database)**: The sanitized Plans are saved directly to local storage (Hive). The app stores exactly **1 Active Diet Plan** and **1 Active Workout Plan** at a time to prevent UI bleeding.

### 6.2. Re-Rolling & Handling Mid-Day Overwrites
When a User generates a *new* AI plan, or a Trainer assigns a *new* Plan:
1. **Database Level**: The backend changes the old plan's status to `ARCHIVED` and sets the new plan to `ACTIVE`.
2. **Target Scheduling (The Mid-Day Trap)**: All newly generated plans MUST have their `startDate` configured to execute starting **Tomorrow**. This prevents UI tearing if a user generates a new plan at 2:00 PM after having already logged their Breakfast under the old plan structure.
3. **Client Level**: The mobile app's next `sync/down` retrieves the newly queued Active Plan, mapping safely out of the current day's active progress logs.

### 6.3. Timezone Conversion (Server vs Mobile Timing)
- **Problem**: Plans generated at 11 PM locally might fall on tomorrow's date based on server timezone geometry.
- **Solution**: The backend strictly saves the `startDate` of ALL plans as `00:00:00 UTC`.
- **Conversion on Mobile**: The mobile app casts this UTC Midnight to the **local device timezone (`.toLocal()`)**. Therefore, `"MONDAY"` maps perfectly to the User's local Monday regardless of geographic latency.

### 6.4. Day Control Matching (AI to Calendar)
- Because the backend formats the AI days to strict Enums (`MONDAY`, `TUESDAY`, etc.), there is no guesswork. 
- The mobile frontend gets the current `DateTime()`, runs `.weekday`, and fetches exactly the `MasterMeal` or `MasterRoutine` aligned exactly to the matching calendar day.

### 6.5. Mobile Sub-Systems: Offline Fallback
- **Offline Fallback**: If the backend completely times out, `ai_orchestration_service.dart` features a failsafe `AIStrategy.offline` mechanism. It will auto-assemble a highly basic "Mock" routine sourced from localized memory variables so the user never sees an endless blank screen.

### 6.6. Smart Shopping List Engine & Buy-Tracking
- **Aggregation & Deduplication**: To avoid the AI mapping "Cooked Chicken" and "Diced Chicken" as different products, the app does not rely purely on display strings. It requires a `canonicalName` (e.g., `chicken_breast_raw`) to perfectly deduplicate arrays. It aggregates identical active lines across 4 weeks (e.g., rolling 7 allocations into a single `700g Chicken` requirement).
- **Member Utility (Checking Off Items)**: This isn't just a static display. The mobile UI generates an interactive interface where members can:
  - Bring their phone to the grocery store.
  - See exactly what total ingredients they need to buy for the entire week to succeed.
  - Tap the checkbox next to an item as they put it in their cart.
  - Instantly see the item get marked as "Bought" or crossed out, making it extremely clear *exactly what else they still need to buy*. This drastically increases diet adherence by eliminating the guesswork of meal prep shopping.

### 6.7. Viewing Yesterday, Today, and Tomorrow
- The user's Dashboard bounds to the `SessionProgressNotifier`. Clicking left or right on the calendar causes a localized time-travel query.
- It calculates the target day of the week, locates the appropriate `MasterMeal`/`MasterRoutine` assignment, and compares it to the Historical snapshot.

### 6.8. User Actions (Marking Tasks), State Persistence & History
- **Names & States**: Tasks have explicit lifecycle states (`MealStatus.pending`, `MealStatus.completed`, `MealStatus.skipped` and `WorkoutStatus.completed`).
- **Interaction**: A user taps their *Active Meal* (e.g., Breakfast) to mark it as `COMPLETED`. Or they tap into an *Active Workout* and finish logging sets.
- **Real-Time Calendar Navigation**:
  - When the user navigates from **Monday to Tuesday**, and then back to **Monday**, the app uses the `SessionProgressNotifier`.
  - The marked meals and workouts will **always remain visible and checked off** exactly as they left them.
  - It works seamlessly across refreshes because the marked state is locked into the local Hive Cache (`DailyProgress`) instantly. The UI checks this memory cache on every single scroll/refresh to reconstruct the colorized green/red checkmarks instantly.
- **Tracking Log (`DailyProgress`)**:
  - The exact moment completion occurs, `session_progress_provider.dart` saves a frozen snapshot in the `DailyProgress` table.
  - Doing this permanently binds the actual calories eaten, protein consumed, and activity tracking metrics directly to that specific Date.
  - `syncService.syncUp()` pushes these local completions to the central backend. Even if the Master Template is generated anew next month, scrolling back to this month's Monday will render the UI perfectly green based on this historical snapshot.

## 📖 7. Mobile Unification Dictionary (JSON Key Schema)

If you are modifying the AI Prompt or building a manual Trainer Dashboard export, your JSON variables absolutely **MUST** match these exact keys for the Mobile Flutter App (`ai_orchestration_service.dart`) to parse them without crashing.

### 7.1 Diet Plan Keys
To achieve **Zero-Hallucinations**, no synonyms or fallbacks are allowed. The AI and Trainer Dashboard MUST output these exact keys:
- **Top Level**:
  - `start_date`: String (e.g., "2026-04-07", optional).
- **Meal Object**:
  - `name`: String (e.g., "Avocado Toast"). *(Strictly forbade `item`)*
  - `time`: String (e.g., "08:00").
  - `type`: String (e.g., "BREAKFAST", "LUNCH", "DINNER").
  - `instructions`: String. *(Strictly forbade `description`)*
  - `prepTimeMinutes`: Integer (e.g., 15).
  - `calories`: Integer. *(Strictly forbade `totalCalories`)*
  - `protein`, `carbs`, `fats`: Integer. *(Strictly forbade `fat`)*

- **Ingredient Object** (inside `ingredients` array):
  - `name`: String (e.g., "Avocado"). *(Strictly forbade `item`)*
  - `canonicalName`: String (e.g., "avocado_raw"). **CRITICAL:** Forces AI to categorize the item correctly for aggressive Shopping List deduplication without breaking due to adjectives.
  - `amount`: Integer (e.g., 100). *(Strictly forbade `grams`)*
  - `unit`: String (e.g., "g", "oz", "ml", "cup"). **CRITICAL:** This must be strictly passed based on the member's chosen EU/US localized metric settings to ensure the Shopping List renders correctly for their region.
  - `calories`, `protein`, `carbs`, `fats`: Integer.

### 7.2 Workout Plan Keys
For workouts syncing down to the `PlannedExerciseEntity`:
- **Routine/Day Object** (`days[]` array):
  - `dayOfWeek`: String enum (`MONDAY`–`SUNDAY`). **CRITICAL**: Must be the canonical full-word enum — never an integer index.
  - `dayName`: String (e.g., `"Push Day Alpha"`). Human-readable session label shown in the mobile UI.
  - `sessionType`: String (e.g., `"STRENGTH"`, `"CARDIO"`, `"REST"`). Used by the mobile to render the correct session icon.
  - `isRestDay`: Boolean.
  - `estimatedDurationMinutes`: Integer.
  - `warmup`: Array of exercise objects (same schema as `mainWork`). May be empty on rest days.
- **Exercise Object** (inside `mainWork` array — **not** `exercises`):
  - `exerciseName`: String.
  - `sets`: Integer.
  - `minReps`: Integer (e.g., 3). Must be absolute minimum 3 for heavy strength profiles.
  - `maxReps`: Integer (e.g., 10). Wave periodization multiplies this value each week.
  - `rpe`: Number (1–10). Rate of Perceived Exertion. Used by mobile to display effort level.
  - `restSeconds`: Integer (e.g., 90).
  - `progressionNote`: String.
  - `muscleGroupPrimary`: String.

*NOTE: The frontend Dart code performs manual numerical sanitization. It will attempt to cast string representation `("15")` into integers `(15)` automatically, but you should explicitly demand INTEGERS from the AI to prevent edge-case parsing crashes.*

## 📊 8. Observability & Monitoring

Every AI job emits a structured `[AI_METRICS]` log line at completion. These are grep-able and feed directly into any log aggregator (Datadog, CloudWatch, Loki).

### 8.1 Key Metrics Emitted Per Job

```json
{
  "userId": "...",
  "type": "WORKOUT | DIET",
  "provider": "OPENAI | ANTHROPIC | DEEPSEEK",
  "model": "gpt-4o | claude-3-5-sonnet | deepseek-chat",
  "totalDurationMs": 4230,
  "fallback_used": false,
  "repair_attempted": true,
  "prompt_tokens": 1820,
  "completion_tokens": 2340
}
```

### 8.2 Ghost Calorie Correction Events

When the diet validator detects a Ghost Calories discrepancy (>50 kcal difference between AI-claimed and ingredient-summed calories), a second `[AI_METRICS]` line is emitted:

```json
{
  "userId": "...",
  "type": "DIET",
  "ghost_calories_corrected": true,
  "ai_claimed": 2800,
  "corrected_to": 2120
}
```

### 8.3 Operational Health Signals

Monitor these 5 rates in your log aggregator to detect system degradation before users notice:

| Metric | Query | Alert Threshold |
|--------|-------|----------------|
| Fallback rate | `fallback_used: true` / total jobs | > 10% over 1h |
| Repair rate | `repair_attempted: true` / total jobs | > 20% over 1h |
| Ghost calorie rate | `ghost_calories_corrected: true` / DIET jobs | > 15% over 1h |
| Avg job duration | mean of `totalDurationMs` | > 10 000ms |
| Token spike | sum of `completion_tokens` per hour | > provider rate limit |

A rising fallback rate signals provider instability or prompt drift. A rising repair rate signals schema drift (AI output no longer matching the system prompt schema). Both are early warnings before user complaints arrive.
